# الخطة النهائية — منصة تعليمية (سنتر متعدد المدرسين)

> **الحالة:** معتمدة للتنفيذ  
> **المكدس:** Laravel 12 + MySQL (Monolith Modular + Service Pattern)  
> **النطاق:** سنتر تعليمي فيه أكثر من مدرس، مع لوحات لكل من: طالب · مدرس · ولي أمر · أدمن  
> **الإصدار:** 1.3 — تبسيط الهوية: بدون موافقة أدمن، كتالوج مدرسين عام، بروفايل مدرس غني، اسم المنصة **سنتر**

---

## قرارات معمارية محسومة قبل يوم التنفيذ الأول

> هذه القرارات **ثابتة من المرحلة 0** وتُصمَّم الجداول والمنطق على أساسها. لا تُؤجَّل لمرحلة التنفيذ اللاحقة.

| # | البند | القرار المعتمد |
|---|---|---|
| 1 | **الدفع** | **فودافون كاش** أو **كاش يدوي** فقط في الوضع الحالي. لا بوابة إلكترونية الآن. دورة: طلب/تسجيل دفع → إثبات اختياري → مراجعة (**أدمن أو مدرس حسب النطاق**) → تأكيد/رفض → تفعيل اشتراك. Schema من المرحلة 0 مع `provider` / `channel` / `recorded_by` / `reviewed_by`. |
| 1b | **الدخول والموافقات** | الدخول **بالبريد فقط**. تسجيل طالب/مدرس/ولي أمر → **نشط فورًا** (بدون موافقة أدمن). **المدرس بوابة المجموعة:** إضافة طلاب من السنتر، قبول طلبات الانضمام، تسجيل كاش (غالبًا نهاية الشهر)، مراجعة فودافون كاش. ربط ولي الأمر بكود الطالب → **نشط فورًا**. الأدمن: هيكل + تقارير + إيقاف/إخفاء من الكتالوج. كتالوج عام `/teachers` قبل التسجيل. |
| 2 | **مزود الفيديو** | **Bunny.net Stream** — Token Authentication / Signed URLs. ليس YouTube وليس تخزينًا محليًا في Production. |
| 3 | **حماية المحتوى من التحميل** | **مطلوب:** Signed URLs قصيرة العمر، بدون زر تحميل، عدم كشف الرابط الخام، إعدادات Bunny ضد hotlinking. PDF قابل للتحميل فقط إن صُرّح لكل مرفق. |
| 4 | **Autosave للامتحان** | **إلزامي:** حفظ كل إجابة فورًا + استئناف بعد انقطاع النت. |
| 5 | **الفروع (Multi-branch)** | **محتمل** → `branch_id` من المرحلة 0؛ فرع واحد افتراضي؛ UI متعدد الفروع مؤجل. |
| 6 | **خدمة الإيميل** | **Mailgun** لـ Staging/Production. Local: `log` / Mailpit. |
| 7 | **لوحة الأدمن** | **بدون Filament.** Blade + Livewire مخصص. |
| 8 | **مكتب المدرس (Teacher Desk)** | لوحة المدرس = **نظام صغير مستقل ضمن صلاحياته**: إضافة طلاب يدويًا، تسجيلهم على مواده، تسجيل مصروفاتهم/مدفوعاتهم (كاش أو فودافون)، تأكيد دفع طلابه، ومتابعة الحسابات — مع عزل صارم: المدرس يرى ويدير **طلابه ومواده فقط**. الأدمن يرى الكل ويمكنه التدخل. |

النقاط التي تبقى كما هي (لا تُمس):
- عدم فصل الفرونت الآن + طبقات API-ready
- Definition of Done لكل مرحلة + لا Feature بدون اختبار
- MVP قابل للتقليص بوضوح

## قرار الواجهة

### السؤال: React منفصل ولا Laravel؟

| الخيار | التوصية |
|---|---|
| **Laravel Blade + Livewire** لكل الأدوار بما فيها الأدمن | **معتمد** |
| Inertia + Vue/React | لاحقًا إن احتجنا تفاعل أثقل |
| React SPA منفصل | **مؤجل** بعد استقرار الـ Core و `/api/v1` |

### القرار النهائي

1. النظام كامل داخل Laravel — واجهات احترافية Blade + Livewire.
2. الطبقات **API-ready** (Services / DTOs / Policies) لفتح موبايل/React لاحقًا بدون إعادة منطق العمل.
3. لا React منفصل ولا موبايل الآن.
4. **لا Filament** — أدمن مخصص Livewire.

### واجهات الأدوار

| الدور | التقنية | الهدف |
|---|---|---|
| **Admin** | Blade + Livewire (لوحة مخصصة) | مستخدمين، هيكل أكاديمي، كل المدفوعات، اشتراكات، تقارير شاملة، إعدادات السنتر |
| **Teacher** | Blade + Livewire (**مكتب مدرس**) | إضافة طلاب، تسجيل مصروفات/مدفوعات، محتوى، بنك أسئلة، امتحانات، متابعة أداء وحسابات طلابه |
| **Student** | Blade + Livewire | دروس (بث Bunny)، امتحانات مع autosave، تقدم، رفع إثبات دفع |
| **Parent** | Blade + Livewire | متابعة الأبناء، الدرجات، حالة الاشتراك/المصاريف |

تصميم UI: هوية واحدة، RTL عربي، متجاوب، فصل بصري واضح لكل دور.

## 1. نظرة عامة على المنتج

منصة ويب لسنتر تعليمي متعدد المدرسين:

- **الطالب:** يتصفح حسب مرحلته/صفه، يشاهد دروسًا محمية، يحل امتحانات (مع حفظ فوري)، يتابع تقدمه ويقدّم إثبات دفع فودافون كاش أو يدفع كاش في السنتر/عند المدرس.
- **المدرس (نظام صغير داخل اللوحة):** يضيف طلابه يدويًا، يسجّلهم على مواده، يسجّل مصروفاتهم ومدفوعاتهم (كاش/فودافون) ويؤكّدها، يدير المحتوى والامتحانات، ويتابع الأداء والحسابات — كإدارة مجموعة سنتر صغيرة بدون انتظار الأدمن لكل عملية يومية.
- **ولي الأمر:** يتابع أبناءه (درجات، تقدم، اشتراكات/مدفوعات).
- **الأدمن:** الهيكل الأكاديمي، التقارير، إشراف المدفوعات، إيقاف حسابات أو إخفاء مدرس من الكتالوج — **ليس** بوابة موافقة يومية على التسجيل.

**نموذج العمل:** سنتر واحد (مع جاهزية `branch_id`) → عدة مدرسين → كل مدرس يدير مجموعة طلابه ومواده ومصروفاتهم → (+ ولي أمر).

**المعمارية:** Monolith Laravel modular:  
`Controller → FormRequest → Service → Repository → Model`  
+ DTOs، API Resources، Policies، Events/Jobs.

## 2. المبادئ المعمارية (ثابتة لكل Module)

### 2.1 الطبقات

- **Controller:** استقبال الطلب فقط.
- **FormRequest:** Validation فقط.
- **Service:** منطق العمل.
- **Repository + Interface:** وصول البيانات.
- **DTO / Policy / Events / Jobs / API Resources:** كما في المعايير العامة.

### 2.2 هيكل المجلدات

```
app/Modules/{ModuleName}/
  ├── Http/Controllers/
  ├── Http/Requests/
  ├── Http/Resources/
  ├── Services/          (+ Interfaces)
  ├── Repositories/      (+ Interfaces)
  ├── DTOs/
  ├── Models/
  ├── Policies/
  ├── Events/ + Listeners/
  ├── Jobs/
  ├── Livewire/          (واجهات الدور إن لزم)
  ├── Providers/{Module}ServiceProvider.php
  └── Routes/web.php     (+ api.php عند الحاجة)
```

الافتراضي: تنظيم يدوي تحت `app/Modules` (بدون nwidart في البداية).

### 2.3 معايير من اليوم الأول

- [ ] Interfaces لـ Services/Repositories الرئيسية
- [ ] Policies لكل Model حساس
- [ ] استجابة API موحّدة + `/api/v1`
- [ ] Soft Deletes: users، نتائج، subscriptions، payments
- [ ] `branch_id` على الجداول المتفق عليها من الـ ERD
- [ ] **اختبار بعد كل Feature** — لا إغلاق بدون اختبارات خضراء

### 2.4 فصل الموديولات

التواصل بين الموديولات عبر Services و Events فقط — لا استدعاء Repository عبر الحدود.

## 3. الموديولات (النطاق المعتمد)

### M1 — Identity & Access
- تسجيل/دخول: Student, Teacher, Parent, Admin
- تحقق بريد (Mailgun) / هاتف OTP لاحقًا إن لزم
- إعادة كلمة المرور
- RBAC — Spatie Permission
- Profiles + ربط Parent ↔ Students
- **إنشاء طالب بواسطة المدرس:** المدرس يضيف طالبًا يدويًا (اسم، هاتف، صف، …) فيُنشأ حساب `student` مع كلمة مرور مؤقتة/إرسال دعوة؛ يُسجَّل `created_by` = المدرس
- سياسة: المدرس يعدّل/يرى الطلاب المرتبطين بمواده أو الذين أنشأهم؛ الأدمن يرى الكل
- 2FA للأدمن قبل الإطلاق الكامل

### M2 — Academic Structure
- Stages → Grades → Subjects → Units/Chapters
- Teacher ↔ Subjects، Student ↔ Grade
- جدول `branches` + فرع افتراضي؛ `branch_id` على التسجيلات/الاشتراكات/المدفوعات
- إعدادات السنتر (اسم، رقم فودافون كاش للتحويل، تعليمات الدفع)
- إعدادات اختيارية لكل مدرس: رقم محفظة فودافون خاص بالمدرس (إن اختلف عن رقم السنتر) — يُعرض لطلابه عند الدفع

### M3 — Content & Lessons
- دروس: نص، **فيديو عبر Bunny Stream** (video_id + إعدادات الحماية)، PDF/مرفقات
- ترتيب داخل الوحدة
- Progress tracking
- حماية بث: Signed URL / token عند المشاهدة فقط لمن له اشتراك فعّال
- تعليقات الدرس: مؤجلة عن MVP إن ضاق الوقت

### M4 — Exams & Assessments
- بنك أسئلة (MCQ, T/F, مقال, إكمال)
- امتحانات: درجات، وقت، محاولات، جدولة
- **Autosave:** حفظ كل إجابة فورًا + استئناف المحاولة
- تصحيح تلقائي + يدوي للمقالي
- نتائج وتقارير
- Assignments كامتداد لاحق لنفس الموديول إن لزم
- حماية: عشوائية، عدم تسريب الإجابات، Rate limit على الحفظ والتسليم

### M5 — Enrollment & Subscriptions
- تسجيل في مادة/كورس بواسطة: الطالب (طلب) أو **المدرس يدويًا** أو الأدمن
- خطط: شهري / فصلي / سنوي / باقة (ومبالغ شهرية بسيطة إن لزم لمكتب المدرس)
- حالات: active / expired / suspended / pending_payment
- التفعيل بعد **تأكيد الدفع** من المدرس (لنطاق طلابه/مواده) أو الأدمن
- ربط واضح: `subscription` ↔ `subject`/`teacher` حتى تُعزل حسابات كل مدرس

### M6 — Payments & Billing (فودافون كاش / كاش) + مصروفات مكتب المدرس
- **قنوات الدفع الحالية فقط:**
  - `vodafone_cash`: رقم العملية و/أو صورة إثبات بعد التحويل لرقم المحفظة (سنتر أو مدرس).
  - `cash`: تسجيل نقدي في السنتر أو عند المدرس.
- **من يسجّل الدفع؟**
  - الطالب: يرفع إثبات فودافون → `pending_review`
  - **المدرس:** يسجّل مصروف/دفعة لطالبه يدويًا (كاش أو فودافون) من مكتبه — كدفتر حسابات مجموعة
  - الأدمن: يسجّل أو يراجع أي دفعة
- **من يؤكّد؟**
  - المدرس: يؤكّد/يرفض مدفوعات **طلابه المرتبطين بمواده فقط** (ومن ضمنها ما سجّله بنفسه كاشًا يمكن تأكيده مباشرة)
  - الأدمن: صلاحية كاملة + إمكانية تصحيح أي سجل
- دورة الحياة: `pending_review` → `confirmed` | `rejected`
- عند `confirmed`: Event يفعّل الاشتراك ويُصدر/يحدّث الفاتورة
- سجل مصروفات/مدفوعات لكل طالب تحت المدرس: المبلغ، السبب/الخطة، التاريخ، القناة، الحالة
- ملخص مالي بسيط للمدرس: محصّل اليوم/الشهر، معلّق، متأخرون
- Invoices + سجل معاملات (`recorded_by`, `reviewed_by`, `teacher_id` للنطاق)
- Coupons / Refunds: بعد استقرار الـ MVP
- تنبيه قرب انتهاء الاشتراك (Mailgun + In-App)
- **ملاحظة تصميم (مرحلة 0):** `channel`, `provider` (= `manual`), `external_reference`, `proof_path`, `recorded_by`, `reviewed_by`, `reviewed_at`, `rejection_reason`, `teacher_id` — جاهزية لبوابة لاحقًا دون هدم الجدول

### M7 — Notifications
- In-App + Email عبر **Mailgun**
- SMS/WhatsApp اختياري لاحقًا
- تذكير امتحان / انتهاء اشتراك / نتيجة مراجعة دفع — عبر Queue

### M8 — Reports & Analytics
- Dashboards لكل دور
- أدمن: اشتراكات، إكمال دروس، متوسط درجات، كل المدفوعات
- **مدرس:** عدد طلابه، تحصيل مصروفاته، متأخرون، أداء امتحانات مجموعته

### M9 — Certificates (بعد MVP)
- PDF + رقم تحقق / QR

### M10 — Media
- مرفقات غير الفيديو: S3-compatible من Staging+
- الفيديو: **Bunny Stream فقط** في Prod
- Signed URLs للمحتوى المحمي

### M11 — Admin / CMS (Livewire مخصص)
- إدارة كل الموديولات من لوحة أدمن Blade/Livewire
- طابور مراجعة كل المدفوعات + إشراف على ما أكّده المدرسون
- صفحات ثابتة: من نحن، FAQ، شروط، تعليمات الدفع

### M12 — Teacher Desk (مكتب المدرس — نظام صغير)
> جزء أساسي من تجربة المدرس، يُنفَّذ عبر M1 + M5 + M6 بواجهات Livewire تحت `/teacher/...` — ليس موديولًا منفصلًا بالضرورة، لكن **متطلب منتج صريح**.

القدرات الإلزامية:
1. إضافة طالب يدويًا + ربطه بمادة/خطة المدرس
2. قائمة طلاب المدرس مع بحث/فلترة
3. تسجيل دفعة/مصروف (كاش أو فودافون) لطالب
4. تأكيد أو رفض إثبات دفع يخص طلابه
5. تفعيل/تعليق اشتراك الطالب على مادة المدرس بعد الدفع
6. كشف حساب مبسّط لكل طالب + ملخص تحصيل للمدرس
7. عزل بيانات: لا يرى طلاب أو مدفوعات مدرس آخر

## 4. مراحل التنفيذ (Milestones)

كل مرحلة = تسليم قابل للتجربة + اختبارات خضراء.  
**التقديرات أدناه تشمل buffer ≈ ×1.3** بعد مراجعة الحمل الحقيقي (4 واجهات Livewire + اختبار إجباري + امتحانات مع autosave).

### المرحلة 0 — Foundation (**~2.5–3 أسابيع**)
- بيئات Local / Staging / Production
- هيكل `app/Modules` + bindings
- Git: `main` + `feature/*`
- MySQL + Migrations أولية
- **ERD كامل** يشمل: `branches`, مدفوعات فودافون/كاش مع `teacher_id`/`recorded_by`, حقول Bunny على `lessons`, `exam_answers` للـ autosave, `created_by` للطلاب
- Spatie Permission + أدوار: `admin`, `teacher`, `student`, `parent` + صلاحيات مكتب المدرس (`students.create`, `payments.record`, `payments.review` scoped)
- إعداد Mailgun (Staging) + توثيق مفاتيح Bunny (حساب + مكتبة Stream)
- CI: phpunit على Push
- Redis لـ cache/queue/session في Staging+
- Layout أساسي موحّد للوحات الأربع (بدون صفحات أعمال بعد)
- **DoD:** هيكل + Seed أدوار + فرع افتراضي + smoke test + `docs/ERD.md` معتمد (يشمل Teacher Desk)

### المرحلة 1 — Identity & Access (**~2–2.5 أسابيع**)
- Auth + Profiles + Parent–Student
- Policies أساسية + هيكل لوحة أدمن/مدرس Livewire (shell)
- **إضافة طالب من لوحة المدرس** (إنشاء حساب + ربط أولي)
- **DoD:** دخول لكل دور + عزل صلاحيات + مدرس يضيف طالبًا ضمن نطاقه فقط (مختبر)

### المرحلة 2 — Academic Structure (**~1.5 أسبوع**)
- CRUD Stages/Grades/Subjects/Units مع مراعاة `branch_id` حيث يلزم
- ربط مدرس↔مادة وطالب↔صف
- **DoD:** أدمن يدير الهيكل؛ مدرس يرى مواده فقط

### المرحلة 3 — Content & Lessons (**~3–4 أسابيع**)
- دروس + ترتيب + Progress
- تكامل رفع/ربط فيديو Bunny + تشغيل محمي للطالب المشترك
- مرفقات (Local → S3 في Staging)
- **DoD:** مشاهدة محمية + تتبع تقدم + مدرس يدير محتوى مادته

### المرحلة 4 — Exams (**~5–6 أسابيع**)
- بنك أسئلة + إنشاء/جدولة + محاولات
- **Autosave لكل إجابة** + استئناف بعد انقطاع
- تصحيح تلقائي/يدوي + تقارير
- حماية تسريب الإجابات + Rate limit
- **DoD:** دورة امتحان كاملة مع autosave مختبرة (بما فيها محاكاة انقطاع)

### المرحلة 5 — Enrollment & Payments + Teacher Desk مالي (**~3–3.5 أسابيع**)
- خطط + اشتراك بحالة `pending_payment`
- تسجيل طالب على مادة المدرس يدويًا من المكتب
- تدفق فودافون كاش (إثبات) + كاش (تسجيل مدرس/أدمن)
- طابور مراجعة المدرس لطلابه + إشراف أدمن
- كشف حساب طالب + ملخص تحصيل للمدرس
- فواتير أساسية
- **DoD:** مدرس يضيف طالبًا → يسجّل دفعة كاش → اشتراك ينشط؛ ومسار إثبات فودافون → تأكيد/رفض — كلها مختبرة مع عزل بين مدرسين

### المرحلة 6 — Notifications & Reports (**~2.5 أسابيع**)
- In-App + Mailgun (مراجعة دفع، تذكير اشتراك، امتحان)
- Dashboards خفيفة لكل دور
- **DoD:** إشعارات حرجة تصل؛ KPIs أساسية صحيحة

### المرحلة 7 — Certificates & Media hardening (**~1.5 أسبوع**)
- شهادات إن ضمن النطاق
- مراجعة Signed URLs + إعدادات Bunny النهائية لـ Prod

### المرحلة 8 — QA الشامل (**~3 أسابيع**)
- Unit للـ Services الحرجة
- Feature لكل Module
- اختبار autosave تحت انقطاع متكرر
- مراجعة أمنية (AuthZ، تسريب امتحان، كشف روابط فيديو)
- UAT مع عينة حقيقية

### المرحلة 9 — Production Readiness (**~1.5–2 أسابيع**)
- Checklist القسم 5
- Backups + Sentry + نشر Staging→Prod

### المرحلة 10 — Launch
- Soft launch → Full launch + دورة Bugfix أول أسبوعين

**التقدير الإجمالي الواقعي:** حوالي **5.5 – 7 أشهر** لفريق صغير–متوسط (بدل 4–5 المتفائلة سابقًا)، مع إمكانية الضغط بتقليص MVP أدناه.

### MVP للإطلاق الأول (تقليص عند ضغط الوقت)

**إلزامي:**

## 7. استراتيجية الاختبار (إجبارية بعد كل Feature)

### القاعدة الذهبية
لا يُغلق Feature إلا بعد: طبقات كاملة + Policy إن لزم + **اختبار واحد على الأقل** + مجموعة الموديول خضراء.

### أنواع الاختبارات

| النوع | يغطي | أين |
|---|---|---|
| **Unit** | Services (تصحيح، تأكيد دفع، تفعيل اشتراك) | `tests/Unit/Modules/...` |
| **Feature** | HTTP/Livewire + AuthZ | `tests/Feature/Modules/...` |
| **Browser** | مسارات حرجة لاحقًا | Dusk عند الحاجة |

### قالب دورة الـ Feature

```
1. Migration + Model
2. Repository + Service + Policy
3. Controller / Livewire (حسب الدور)
4. Feature Test: happy path + unauthorized + validation fail
5. php artisan test --filter=FeatureName
6. حدّث docs إن لزم
7. ادمج الـ PR
```

### حد أدنى قبل كل Milestone
- Auth: login + عزل الأدوار
- Academic: CRUD + عزل مدرس
- Lessons: progress + رفض مشاهدة بدون اشتراك
- Exams: attempt + **autosave** + auto-grade MCQ + منع تسريب إجابات + استئناف بعد انقطاع
- Teacher Desk: مدرس أ يضيف طالبًا ويسجّل دفعة → ينجح؛ مدرس ب **لا** يرى الطالب ولا الدفعة
- Payments: إثبات فودافون → تأكيد مدرس النطاق → اشتراك نشط؛ ومسار رفض؛ أدمن يراجع الكل

## 9. قرارات نطاق مثبتة (مرجع سريع)

| البند | القرار |
|---|---|
| ولي الأمر | داخل النطاق من المرحلة 1 |

## 11. ملخص تنفيذي

منصة سنتر متعدد المدرسين على Laravel Monolith modular، واجهات Livewire لكل الأدوار **بدون Filament**. كل مدرس لديه **مكتب صغير**: يضيف طلابه، يسجّل مصروفاتهم ومدفوعاتهم (فودافون كاش/كاش)، ويؤكّدها ضمن نطاقه — مع إشراف أدمن على الكل. فيديو **Bunny Stream** محمي، امتحانات بـ **autosave**، `branch_id` و**Mailgun** من المرحلة 0، واختبار بعد كل Feature إلزامي (خصوصًا عزل بيانات المدرسين).
